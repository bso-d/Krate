How the observability layer fits

One self-contained stack per cluster. The exporter joins the running cluster's Docker network and reads metrics as an ordinary Kafka client — no JMX java-agent, no broker restart. Nothing here reaches outside the host.

Source ZK cluster 4 brokers · ZooKeeper kafka-92…95:909x

$ kafka monitor up # exporter + Prometheus + Grafana, joined to the cluster net

alerts/

Alert rules — dynamic, not hardcoded

Rules adapt to whatever the cluster size is. "Broker disappeared" compares the live count to the last hour's peak, so it needs no fixed broker number baked in.

AlertConditionForSeverity
KafkaExporterDown up{job="kafka"} == 0 2m Critical
KafkaBrokerDisappeared kafka_brokers < max_over_time(kafka_brokers[1h]) 2m Critical
UnderReplicatedPartitions sum(...under_replicated_partition) > 0 5m Warning
ConsumerGroupHighLag sum by (consumergroup)(...lag) > 1000 5m Warning

Producer / consumer path

Topic orders — 6 partitions, RF 3 — took 200 messages; a group consumed 50, leaving lag 150 that surfaced identically in the exporter, Prometheus, and Grafana.

Two bugs fixed to get here

kafka lag called kafka-consumer-groups.sh (absent in cp-kafka 7.6.1); the proxy healthcheck probed localhost (IPv6) while nginx bound IPv4. Both corrected in the zk and kraft CLIs.

freeze/

The line: this is where ZooKeeper stops

❄ Frozen — no new features

The ZooKeeper edition is complete and stays as-is. Bug and security fixes only. It remains a valid, fully observable target for air-gapped hosts that still run a ZK-based Kafka.

Forward work lives in KRaft

The consumer-groups and healthcheck fixes were mirrored into the kraft CLI. New capability — sizing, topic scaffolding, DR/failover — belongs to the KRaft edition, not here.
